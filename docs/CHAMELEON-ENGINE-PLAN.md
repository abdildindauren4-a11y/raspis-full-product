# РАСПИС Engine v2 «Хамелеон» — толық құру жоспары

> Мақсат: кесте алгоритмін кез келген мектептің ережелеріне бапталатын,
> жеке әрі толыққанды **екінші қозғалтқыш** ретінде құру. v1 (қазіргі
> `src/algorithm/engine.ts`) жұмысын жалғастыра береді — v2 дайын болып,
> сапасы дәлелденгенде ғана ауысамыз.

---

## 1. Неге бөлек алгоритм (патч емес)

v1-де ережелер алгоритмнің ішіне тігілген: `hardCheck` ішіндегі шарттар,
greedy кезектің приоритеттері, swap фазаларының логикасы — бәрі бір
файлда, бір-бірімен өрілген. Жаңа ереже қосу оңай (бір шарт), бірақ
ережені **қолданушыға басқартқызу** (өшіру/өзгерту/салмағын беру) — мүмкін
емес: конструкция бұған арналмаған.

v2-нің басты идеясы — **НЕ (ережелер) мен ҚАЛАЙ (шешуші) бөлінеді**:

- Ереже = дерек (объект): аты, түсіндірмесі, түрі, параметрлері, қосулы ма
- Шешуші (solver) = ережелер тізімін оқитын әмбебап механизм
- Мектеп ережелер жинағын өзі басқарады → алгоритм «хамелеон»

## 2. Архитектура

```
src/algorithm2/
├── time.ts          — уақыт моделі: аптадағы күн (5/6), күндегі слот (8-10),
│                      ауысымдар, қоңырау кестесі
├── model.ts         — Slot, Placement, ScheduleState (матрицалар, индекстер)
├── rules/
│   ├── types.ts     — Rule интерфейсі (төменде), RuleContext
│   ├── registry.ts  — барлық ереженің реестрі + мектеп конфигін қосу
│   ├── hard/*.ts    — қатаң ережелер (әрқайсысы жеке файл)
│   └── soft/*.ts    — жұмсақ ережелер (айыппұл функциялары)
├── solver/
│   ├── seed.ts      — бастапқы орналастыру (greedy, приоритет кезегі)
│   ├── repair.ts    — сыймағанды жөндеу (кедергіні жылжыту)
│   ├── improve.ts   — жақсарту (swap: maximin, жайлылық, теңгерім)
│   └── search.ts    — терең іздеу (multi-seed) және ішінара режим (partial/anchor)
├── explain.ts       — «неге болмайды»: әр қабылданбаған орынның себебі
├── profiles.ts      — дайын жинақтар: «Стандарт ҚР», «Гимназия», «Шағын мектеп»
└── index.ts         — generate2(input, config): v1-мен бірдей нәтиже пішімі
```

### Қозғалтқыш = модель (ЖИ-чаттардағыдай ауыстырғыш)

Сайт ішінде алгоритмдер **модельдер** ретінде тұрады — ChatGPT/Claude-тағы
модель таңдағышқа ұқсас:

- **Ауыстырғыш UI:** Генерация және Алгоритм беттерінің жоғарғы жағында
  ашылмалы модель тізімі: аты, белгісі, бір жолдық сипаттамасы —
  «РАСПИС Классик (v1) — тұрақты, дәлелденген» /
  «РАСПИС Хамелеон (v2) — икемді, ережелері бапталады». Бір басу — ауысты.
- **Баптаулар модельге тіркелген:** әр модельдің ӨЗ конфиг кеңістігі бар —
  `engineConfigs: { v1: {…қазіргі алгоритм баптаулары}, v2: {ережелер
  конфигі} }` (dataStore-да, бұлтқа синхрондалады). Модель ауысқанда
  Алгоритм бетінің мазмұны да сол модельдің баптауларына автоматты
  ауысады; ешбір баптау жоғалмайды — әр модельдікі өзінде сақтаулы.
- **Нұсқаларда із:** әр сақталған кесте қай модельмен құрылғанын жазады
  (`engine: "v1" | "v2"`) — сапа есебі, ақылды жаңарту (partial/anchor)
  сол модельмен жүреді.
- **Реестр кеңеймелі:** модельдер тізімі кодта бір массив — болашақта
  үшінші модель (мыс. эксперименттік) қосу да бір жазба.

### Ереже интерфейсі (жүйенің жүрегі)

```ts
interface Rule {
  id: string;                 // "one-subject-per-day"
  title: LangText;            // UI-да көрінетін аты (kk/ru/en)
  description: LangText;      // қарапайым тілмен түсіндірмесі
  kind: "hard" | "soft";      // қатаң (бұзылмайды) / жұмсақ (айыппұл)
  defaultEnabled: boolean;
  params?: ParamSchema[];     // UI автоматты форма құрады (сан/қосқыш/тізім)
  // Орналастыру алдындағы тексеріс (ыстық жол — жылдам болуы шарт):
  check?(ctx: RuleContext, p: CandidatePlacement): string | null;
  // Дайын кестені бағалау (жұмсақ ережелер салмағы):
  penalty?(state: ScheduleState): number;
}
```

Шешуші ережелердің МАЗМҰНЫН білмейді — тек тізімін жүреді. Жаңа ереже
қосу = бір файл жазу + реестрге тіркеу. Мектеп конфигі = `{ruleId:
{enabled, params, weight}}` — dataStore-да сақталады, бұлтқа синхрондалады,
экспорт/импорт етіледі (мектептен мектепке көшіруге болады).

### Уақыт моделі (v1-дің қатып қалған жерлері)

- `daysPerWeek: 5 | 6` — сенбілік мектептер үшін
- `maxSlotsPerDay: 8..10` — нөлдік сабақ/ұзартылған күн
- Барлық матрица/цикл осы параметрлерден құрылады (қазіргі 5 пен 8
  константалары жойылады)

## 4. Шешуші (solver) стратегиясы

v1-дің ӨЗІН-ӨЗІ ДӘЛЕЛДЕГЕН эвристикалары сақталады — тек ережелерге
қарайтын болады:

1. **Seed:** приоритет кезегі (арнайы кабинет → топ → сағат көп → ауыр) →
   әр тапсырмаға ең жақсы (hard өткен, soft ұпайы max) орын
2. **Repair:** сыймағандарға кедергіні жылжыту (v1 ЭТАП 5.5)
3. **Improve:** swap циклдары — maximin + жұмсақ ережелердің жалпы
   айыппұлын азайту (салмақтар конфигтен)
4. **Search:** терең іздеу (multi-seed, ең жақсысын таңдау) және partial/
   anchor (ақылды жаңарту) — v2-де де толық жұмыс істейді

**Өнімділік бюджеті:** реестр арқылы өту қымбаттамауы үшін — генерация
басында қосулы ережелер **жалпақ массивке компиляцияланады** (индирекция
жоқ, аллокация жоқ). Мақсат: v1-ден ≤ +20% (33 сыныпта ≤ 250 мс).

## 5. Кезеңдер (milestones)

### M1 — Ядро + паритет негізі ✅ (2026-07-13 орындалды)
time.ts, model.ts, Rule интерфейсі, реестр, 6 базалық қатаң ереже,
seed-фаза. **Өлшем:** кіші мектепте конфликтісіз кесте құрады.
*Нәтиже: seed мектебінде (22 сынып, 532 сағат) 0 конфликт, 0 орналаспаған,
64 мс; ережені өшіру/параметрлеу жұмыс істейді; конфликт ережелері
өшірілмейтіні тексерілді.*

### M2 — Толық паритет v1-мен 🔄 (басым бөлігі дайын, 2026-07-13)
Барлық v1 ережесі реестрде, repair/improve/search көшірілді, partial/anchor
жұмыс істейді. **Өлшем:** A/B тест — seed және bigSeed деректерінде сапа,
тесік, орналаспаған сандары v1-ден кем емес; уақыт бюджет ішінде.
Осы кезден generate2 «көлеңкеде» жүреді (нәтиже салыстырылады, қолданушыға
v1 көрсетіледі).
*Ағымдағы A/B (паритет-харнесс): seed — сапа 87>86, орналаспаған 0=0,
тесік 1 (v1: 0); bigSeed — сапа 84>76, тесік 0=0, орналаспаған 4 (v1: 3).
Қалды: соңғы тесік/дефицит жабылуы (күндік backtracking), partial/anchor,
уақыт бюджеті (қиын кейсте v1-ден ~2.5x — multi-seed барлауы).*

### M3 — Ережелер конструкторы (UI)
Алгоритм бетінде «Ережелер» бөлімі: тізім, қосқыштар, параметр формалары
(ParamSchema-дан автоматты), салмақ сырғытпалары, «әдепкіге қайтару»,
профильдер (Стандарт/Гимназия/Шағын мектеп), конфиг экспорт/импорт.
Мұғалім/пән карточкаларында апта торы («икстап тастау»).
**Өлшем:** завуч кодсыз кез келген ережені өшіре/баптай алады.

### M4 — Құрылымдық икемділік
6 күндік апта, 10 слотқа дейін, жаңа жеке ережелер (пән/мұғалім/сынып
кестедегі кестелері) UI-мен бірге. **Өлшем:** сенбілік мектептің кестесі
құрылады.

### M5 — Қолмен басқару (drag-and-drop) ⭐
Расписание бетінде сабақты ұстап жылжыту:
- жылжытқанда бос ұяшықтар боялады: жасыл (болады) / қызыл (болмайды)
- қызылға апарса — themes explain.ts себебін айтады: «Ахметова 3-сабақта
  5Б-да», «СанПиН: күндік балл асады»
- сабақты «бекіту» (pin) — генерация оны айналып өтеді (partial locking
  дайын тұр)
- әр қолмен өзгеріс тарихқа жазылады (undo)
**Өлшем:** завуч кестені өз қолымен түзеп, жүйе қателіктен қорғайды.
Бұл — сатудағы ең күшті köрініс: «алгоритм ұсынады, шешім сіздікі».

### M6 — ЖИ-баптаушы чат ⭐
Алгоритм бетінде чат: завуч хамелеонды қалай баптағысы келетінін еркін
мәтінмен жазады («Ахметова сейсенбіде жұмыс істемейді, дене шынықтыру
1-сабаққа қойылмасын, 9-сыныптарға жұма күні 6 сабақтан көп болмасын») —
ЖИ соны ережелер конфигіне аударып, автоматты баптау жасайды.

Қалай жұмыс істейді:
1. Реестрдің ParamSchema-сы промптқа құрал-схема ретінде беріледі —
   ЖИ тек **бар ережелердің** JSON конфигін қайтара алады (ойдан ереже
   шығара алмайды)
2. Gemini кілт-батареясы қайта қолданылады (salesBot-тағы ротация)
3. Жауап реестр схемасына қарсы валидацияланады; аты табылмаған
   мұғалім/пән болса — нақтылау сұрағы қойылады
4. Қолданар алдында адам тілінде алдын-ала көрініс: «Мына 3 ереже
   қосылады: …» + «Қолдану» батырмасы — ЖИ ешқашан көзсіз қолданбайды
5. Әр қолданыс тарихқа жазылады (қайтаруға болады); ЖИ қолжетімсіз
   болса конструктор (M3) қолмен баптауға әрқашан ашық

**Өлшем:** завуч бір хабарламамен 3-5 ережені қолмен формаға тимей-ақ
баптайды. Сату кезінде: «мектебіңіздің ережелерін жай ғана айтыңыз».

### M7 — Ауысу
Модель ауыстырғышта әдепкі модель «Хамелеон (v2)» болады; «Классик (v1)»
тізімде қала береді — қолданушы қалаған кезде бір басып ауыса алады
(ЖИ-чаттардағы ескі модельдер сияқты). ARCHITECTURE.md жаңарады.

Ескерту: модель ауыстырғыштың ӨЗІ M2 соңында-ақ қосылады (v2 «көлеңкеден»
шыққанда) — паритет тесті өткен бетте екі модельді де қолданушыға беруге
болады, «beta» белгісімен.

## 6. Тестілеу стратегиясы

- **Алтын деректер:** seed (22 сынып), bigSeed (33 сынып) + әр нақты
  мектептің анонимделген дерегі қосылып отырады
- **Паритет-харнесс:** екі қозғалтқышты бір деректе жүргізіп, сапа/тесік/
  орналаспаған/уақыт кестесін шығаратын скрипт (CI-ға қосуға болады)
- **Ереже-регрессия:** әр ережеге мини-тест (қосулы → бұзылмайды,
  өшірулі → шектемейді)
- **Fuzz:** кездейсоқ шағын мектептер генерациясы — құламау кепілі

## 7. Тәуекелдер және қарсы шаралар

| Тәуекел | Қарсы шара |
|---|---|
| Реестр индирекциясы баяулатады | ережелерді жалпақ массивке компиляциялау, ыстық жолда аллокациясыз |
| v2 сапасы v1-ден төмен шығады | M2 паритет-қақпасы: өтпейінше ауыспаймыз, v1 өшірілмейді |
| Ережелер UI күрделі көрінеді | әдепкі профиль өзгеріссіз жұмыс істейді; конструктор — «қосымша» экран |
| Екі қозғалтқышты қатар ұстау ауырлығы | v1 мұздатылады (тек қате түзету), барлық жаңа ереже тек v2-ге |
| ЖИ ережені қате түсінеді | схема-валидация + алдын-ала көрініс + «Қолдану» батырмасы + қайтару тарихы — ЖИ тікелей ештеңе өзгертпейді |
| Уақыт | M1-M2 — ең көлемдісі; M3-M5 модульдік, кез келген ретпен |

## 8. Кейінгі көкжиек (жоспардан тыс, бірақ есте)

- Ағындар/біріктірілген топтар (екі сынып бір сабақта)
- Ережелер маркетплейсі: бір мектептің профилін екіншісіне бір батырмамен
