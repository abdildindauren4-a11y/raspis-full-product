# Мобильді приложение дизайнын құру — практикалық нұсқаулық

> NotebookLM дереккөзі. Дизайнер емес, бірақ сапалы өнім жасайтын құрылтайшыға
> арналған. RASPIS/Toi Like тәжірибесінен + салалық стандарттардан жинақталды.

## 1. Негізгі қағидалар (бұзба)
1. **Бір экран — бір мақсат.** Әр экранда бір басты әрекет (primary CTA).
   Пайдаланушы «енді не істеймін?» деп ойланбауы керек.
2. **Mobile-first.** Кіші экранға жаса, сосын үлкенге кеңейт. Тап-аймақ ≥ 44×44px.
3. **Сенім — көрінетін нәрсе.** Нақты фото, пікір, «расталған» белгісі, ашық
   баға. Әр экранда кемінде бір сенім-сигнал.
4. **Таныс болсын.** Дөңгелекті қайта ойлап таппа: төменгі таб-навигация, свайп,
   pull-to-refresh — қолданушы білетін паттерндер.
5. **Жылдамдық сезімі.** Skeleton-жүктеу (spinner емес), оптимистік UI
   (әрекет бірден көрінеді, сервер артынан расталады).

## 2. Дизайн-жүйе (Design System) — ең маңызды инвестиция
Жеке экрандарды емес, **қайта қолданылатын жүйе** жаса. Құрамы:
- **Токендер:** түс, типография, spacing (4px-тор: 4/8/12/16/24/32/48), радиус,
  көлеңке. Барлық жерде БІР мәннен алынады → бірізділік автоматты.
- **Компонент-кітапхана:** Button, Input, Card, Badge, Modal/BottomSheet, Tabs,
  EmptyState... Әрқайсысының күйлері: default/hover/pressed/disabled/loading.
- **Күй-стандарттар:** әр экранда 4 күй міндетті — Loading, Empty, Error, Success.
  Empty ешқашан тұйық емес — келесі-қадам CTA-мен.
- Toi Like мысалы: PRD §27 (түс-токендер, 18 компонент, 6 ереже).

## 3. Түс
- Рөл-негізді: primary (басты әрекет), surface (фон), text (strong/muted),
  success/warning/danger/info (статус). Брендтен бөлек — функционалдық.
- **Статус тек түспен берілмейді:** түс + иконка + мәтін (түс-соқырлыққа).
- Контраст WCAG AA (≥4.5:1). Dark mode — бірінші класс (кешкі қолданыс көп).

## 4. Типография
- 1-2 қаріп жеткілікті (мыс. Inter — кириллица толық). Тақырыпқа жеке қаріп — hero-ға ғана.
- Шкала: H1/H2/H3/Body/Caption — нақты өлшемдермен, жол-биіктігі 1.4-1.5.
- Сандар (баға): tabular-nums; ешқашан қысқартпа («1,4М» емес — «1 400 000 ₸»).

## 5. Навигация мен ақпарат-архитектурасы
- Негізгі бөлімдер ≤ 5 (төменгі таб). Тереңдік ≤ 3 деңгей.
- Артқа-жол әрдайым айқын. Модалдан шығу оңай.
- Пайдаланушы «қайдамын, қайдан келдім, қайда бара аламын» — үшеуін де білсін.

## 6. Формалар (конверсия осында өледі не тірі болады)
- Бір бағанда, label өрістің үстінде, қате-мәтін астында.
- Керек өрісті ғана сұра. Автотолтыру (OTP авто-оқу, қала-геолокация).
- Прогресс көрсет (көп-қадамды формада). Батырма күйі: loading + disabled.

## 7. Микро-өзара әрекет пен эмоция
- Сәтті әрекет — қысқа мереке (конфетти/чек), сосын бірден «келесі не?».
- Кері байланыс лезде (тап → визуалды жауап <100мс).
- Бос күту — «тірі» мәтінмен (RASPIS: «47 күнтізбе тексерілуде...» — күту сенімге айналады).

## 8. Қолжетімділік (accessibility)
Контраст, тап-аймақ ≥44px, экран-оқығышқа белгілер (labels), шрифт-масштабқа
төзімді макет, тек-түске тәуелсіздік. Бұл — «қосымша» емес, сапа.

## 9. Дизайнер емес адамға құралдар
- **Figma** — стандарт (тегін бастау). Дизайн-жүйе + прототип.
- **Refactoring UI** (кітап) — программистке UI-ды жақсартудың нақты ережелері.
- Референс-галереялар: Mobbin, Dribbble (көшірме емес — идея).

## 10. Жиі қателер (RASPIS/Toi Like тәжірибесі)
- Бір экранда 3 primary батырма → пайдаланушы қатып қалады.
- Жасырын дедлайн/таймер → сенімсіздік. Әрқашан көрсет.
- Ақша қысқарту, статус тек түспен → шатасу.
- Дизайн-жүйесіз старт → 30 экран, 30 түрлі стиль, қолдау азабы.
- Bundle-ды оптимизацияламау → әлсіз интернетте алғашқы әсер бұзылады
  (lazy-жүктеу, сурет-оптимизация — RASPIS-те нақты мәселе болды).
