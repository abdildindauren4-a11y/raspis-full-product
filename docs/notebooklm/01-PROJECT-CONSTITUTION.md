# Toi Like — Жоба конституциясы (өзгермейтін қағидалар)

> Бұл — жобаның «өзгермейтін заңдары». Кез келген жаңа мүмкіндік осыларға
> қайшы келмеуі керек. NotebookLM әр архитектуралық аудитте осы файлға
> салыстырады: «жаңа шешім қай баппен қайшы?» Бір бап өзгерсе — ол саналы,
> құжатталған шешім болуы керек (git-те тарихымен).

## I. Миссия
Қазақстанның (кейін Орта Азияның) той/іс-шара индустриясының **операциялық жүйесі**
болу. Клиент 3 минутта дайын команда табады; орындаушы тұрақты тапсырыс алады.

## II. Өнім қағидалары (бұзылмайды)

1. **Каталог емес — құрастырушы.** Басты құндылық = Smart Box (дайын, сол күні
   бос команда). Кез келген фича осы негізгі ағынды күшейтуі керек.
2. **Тесіксіздік аналогы = қос бронь ЖОҚ.** Бір орындаушы-күн ешқашан екі
   клиентке сатылмайды. Транзакциялық құлып — қасиетті (§6.4).
3. **Ақша қорғалған.** Escrow-үлгі (агенттік): клиент төлемі той өткенше
   ұсталады; ledger қос жазба, инвариант Σдебет=Σкредит әрқашан сақталады.
4. **Ашық баға.** Клиент көрген баға = түпкі баға. Комиссия орындаушыдан,
   payout кезінде. Жасырын үстеме — тыйым.
5. **Сенім = өмір.** Verified-only пікір (тек өткен бронь). Premium ақшамен
   сапаны жоғары шығара алмайды — тек тең сапада көтереді.
6. **Тесік/дедлайн жасырылмайды.** Hold/растау таймерлері әрқашан көрінеді;
   бас тарту салдары алдын ала калькулятормен көрсетіледі.
7. **Клиент ешқашан бос қалмайды.** Орындаушы бас тартса — авто-алмастыру
   каскады; табылмаса — толық қайтарым + support.
8. **Өсу — өнімнің ішінде.** Тегін шақыру-қозғалтқыш (§28): маркетинг
   пайдаланушының өзінен туады. Негізгі шақыру мәңгі тегін.

## III. Бизнес қағидалары
- Базалық комиссия 8% (Premium 6%, ресторан 4%, Founding 0%/6ай).
- North Star = аптасына сәтті өткен той саны (Completed Events / Week).
- Supply-бірінші қала-ашылу: тығыздық болмаса — жаңа қала ашылмайды.
- Маусымдылық жоспарға кіреді (сәуір-маусым, тамыз-қараша — шың).

## IV. Техникалық қағидалар (бұзылмайды)
1. **Ақша — BIGINT тиын.** Float ешқашан.
2. **Идемпотенттілік** барлық ақша/бронь операциясында.
3. **Transactional Outbox** — DB-жазу + оқиға бір транзакцияда.
4. **Append-only:** ledger, audit_logs, messages — UPDATE тыйым.
5. **Деректер ҚР-да** («Дербес деректер» заңы) — хостинг ҚР ішінде.
6. **Graceful degradation:** AI/іздеу/WS құласа — өнім жұмысын жалғастырады.
   AI — үстеме қабат, тәуелділік емес.
7. **Модульдік монолиттен бастау** — микросервис тек өлшенген қажеттілікте.
8. **Адам циклде:** қайтымсыз/ірі әрекеттер (ban, ірі refund) — адам растауымен.

## V. Даму процесі (жұмыс тәртібі)
1. Идея → NotebookLM инженерлік промпт жасайды (acceptance criteria,
   тәуелділіктер, қауіпсіздік, өнімділік, масштаб).
2. Промпт → орындаушы AI (Claude Code) кодтайды.
3. **Өлшеу-тәртібі (RASPIS-тен):** диагноз → ӨЛШЕ → түзет → ҚАЙТА ӨЛШЕ.
   Сенімге емес, санға сүйену. LLM сенімді түрде қателеседі.
4. Кезең аяқталғанда → NotebookLM архитектуралық аудит: жаңа шешім бұрынғы
   модульдермен үйлесе ме, болашақ проблема бар ма.

## VI. Шешім қабылдау иерархиясы (қайшылық болса)
Тесіксіздік/ақша-қауіпсіздігі > педагогика/бизнес-ереже > сапа/ыңғайлылық.
(RASPIS сабағы: «тесік болмасын» ережесі басқа ережелерден жоғары тұрды.)
