# RASPIS инженерлік сабақтары — «Қалай жұмыс істейміз»

> Бұл құжат — RASPIS (мектеп кестесін құру SaaS, өндірісте) жобасынан алынған
> нақты инженерлік тәжірибе. Мақсаты екі жақты: (1) NotebookLM-ге «біздің
> инженерлік мәдениетіміз» үлгісі; (2) Toi Like-тағы Smart Box сол алгоритм
> класына жататындықтан — тікелей қолданбалы білім.

## 1. Smart Box = RASPIS кесте-қозғалтқышының ДНҚ-сы
Екеуі де — **шектеулерді қанағаттандыру** (constraint satisfaction):
«шектеулер жиынын қанағаттандыратын үйлесімді комбинация тап».

| RASPIS (кесте) | Toi Like (Smart Box) |
|---|---|
| Сынып × күн × сабақ орналастыру | Той × күн × орындаушы-команда |
| Шектеу: мұғалім бос, кабинет бос, күндік лимит | Шектеу: орындаушы бос, бюджет, қала |
| Ереже: отбасы бөлек күн, қиын-қатар тыйым | Ереже: конфликт-детекция, әртүрлілік |
| Фаза: сүзгі→скоринг→greedy→repair→көп-seed | Дәл сол фазалар (PRD §5.6) |
| hold/rollback (своп сәтсіз → дәл кері) | hold/TTL + каскад-rollback |

## 2. Архитектура-үлгі (engine.ts, ~3300 жол)
Фазалық ЭТАП-құрылым: матрицалар → приоритет-кезек → greedy орналастыру →
repair (кедергіні жылжыт) → maximin (ең нашар сыныпты көтер) → мұғалім-терезе
своптары → апта-баланс → **тесік-толтырғыш кепілдері** → стресс-тесттер → сапа.
Сабақ: **кепілдік қабаттары** маңызды — негізгі алгоритм 95% шешеді, соңғы 5%-ды
арнайы «guarantee» пасстар жабады (Toi Like-та: авто-алмастыру каскады сол рөлде).

## 3. ЕҢ МАҢЫЗДЫ САБАҚ: өлшеу-тәртібі
Осы жобада бірнеше рет **диагнозым қате болды**, тек ӨЛШЕУ түзетті:
- «Мұғалім жетіспейді» дедім → өлшеу: жоқ, менің скриптім қате санаған. Нақты
  себеп басқа болды.
- «Дене шынықтыруды 1-сабаққа қатаң тыйым салам» → өлшеу: бұл ТЕСІК туғызды
  (кейбір күн 4 ауыр+1 жеңіл — ДШ-сыз шешілмейді). Жұмсақ әдіске көштім.
- Класс-рейтинг «46% қызыл» → өлшеу: алгоритм емес, ФОРМУЛА қате (дисперсияны
  жазалап, әдейі жасаған «жеңіл Жұманы» жазалаған).

**Тәртіп (қатаң): гипотеза → ӨЛШЕ → түзет → ҚАЙТА ӨЛШЕ → регрессияны тексер.**
LLM (мен де) сенімді түрде қателеседі. Санға сүйен, сезімге емес.

## 4. Фазз-тестілеу (сенімділікті дәлелдеу)
30 «жасанды мектеп» генераторымен (кіші ауыл → 33 сыныпты 2 ауысым) алгоритмді
жаппай сынадық: крах 0/30. Сабақ: **өз деректеріңмен ғана емес, спектрдің екі
шетімен де сына.** Toi Like: 30 «жасанды сұраныс» (әр қала/бюджет/күн) —
Smart Box генерациясын солай сынау керек.

## 5. Өнімділік — өлшеп оптимизациялау
CPU-профиль көрсетті: уақыттың 60%+ `findRoom`+`cellFree`-де (толық массив
сканері). Индекс қостық (O(N)→O(ұя)): үлкен мектеп 26с→8с. Сабақ: **болжамамен
емес, профильмен оптимизациялау.** Toi Like: Smart Box пул-сұранысы ыстық жол —
дәл солай индекстеу керек (PRD §14.3).

## 6. Тесіксіздік — приоритет иерархиясы
Қайшылық туғанда (тесік vs ереже): тесіксіздік жеңеді. Toi Like конституциясы
VI-бабы дәл осыдан. **Приоритетті АЛДЫН АЛА анықта, әйтпесе әр шеткі жағдайда
шешім тұрақсыз болады.**

## 7. Инфрақұрылым-гигиена (нақты бүліну-багтардан)
- Firestore `undefined` мәнді қабылдамай сақтауды үнсіз құлатқан →
  `ignoreUndefinedProperties`. Сабақ: **үнсіз құлау — ең қауіпті.**
- Синхрон: бұлт үнемі жергіліктіні басып, дерек жоғалған → таймстамп newer-wins.
  Сабақ: **екі жақты синхронда «кім жаңа» ережесі айқын болсын.**
- Кері қайтару багы: клон жасап, ескі сілтеме slots-тан шығып, бөтен элемент
  өшкен. Сабақ: **rollback ДӘЛ болуы керек — түпнұсқа объект, дәл индекс.**
  (Toi Like Saga-компенсацияларына тікелей қатысты.)

## 8. Қауіпсіздік — клиентке сенбе
RASPIS: рөл/тариф/квота бәрі Firestore rules-та (сервер), клиентте емес.
Құжатталған қауіп-модель (privilege escalation, IDOR, quota-farming). Сабақ:
**клиенттегі тексеру = UX, серверлегі = қауіпсіздік. Екеуін шатастырма.**

## 9. Жақсы промпт-үлгілер (осы жобадан, промпт-фабрикаға)
Жақсы тапсырма құрамы: **симптом (нақты) + өлшенген дерек + қайталау-профилі +
күдікті орын + «түзелгенін қалай өлшейміз» критерийі.** Мысал:
> «8Ә-да Сейсенбіде Био+Алг+Ағыл (үшеуі ≥9) қатар тұр — үш-қиын ережесі бұзылған.
> Қай фазада бұзылатынын фаза-фаза бақылап тап, түзет, тек қатар САНЫ азайса ғана
> қабылда (жаңа қатар тудырма), №21 матрицасында регрессия жоқ екенін дәлелде.»
